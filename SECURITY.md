1:  # Security Policy & Vulnerability Reporting
2:  
3:  ## Supported Versions
4:  
5:  We actively maintain and provide security updates for the current active alpha/beta branches. Legacy or unmaintained release forks are not covered.
6:  
7:  | Version | Supported          |
8:  | ------- | ------------------ |
9:  | < v1.0  | :white_check_mark: |
10: 
11: ## Reporting a Vulnerability
12: 
13: **Do not open an unencrypted public GitHub Issue for security bugs.**
14: 
15: If you discover a security vulnerability within `xva-tool` or its modular plugin subsystems, please report it responsibly by using one of the following secure channels:
16: 
17: 1. **GitHub Private Vulnerability Reporting:** Go to the main repository page on GitHub.com, click on the **Security** tab, and select **Vulnerability reporting** to submit a private draft advisory securely.
18: 2. **Direct Encrypted Contact:** Reach out directly to the maintainer via email at **Alexander Maassen <outsider@cuci.nl>**. If applicable, please use a secure, encrypted communication channel or PGP keys to safeguard the vulnerability details during transport.
19: 
20: ### Required Information
21: 
22: To help us triage and resolve the issue efficiently, please include the following details in your report:
23: *   A detailed description of the vulnerability and its potential impact.
24: *   Step-by-step instructions, proof-of-concept (PoC) scripts, or commands required to reproduce the exploit.
25: *   The exact version of `xva-tool`, host operating system kernel, and any active plugin modules involved.
26: 
27: ## Our Commitment
28: 
29: Upon receiving a valid security report, the maintainer will:
30: *   Acknowledge receipt of the vulnerability report within 48 business hours.
31: *   Work closely with the reporter to investigate, isolate, and remediate the issue within a reasonable timeline.
32: *   Coordinate a synchronized public security advisory along with the corresponding patch release to protect production deployments.
